# 阶段性治理流程参考

> **生命周期**：长期维护——跨功能复用的操作参考文档（非规则条款），受 §10 退役评估约束。
>
> **定位**：本文件是**操作参考**，不是规则。下次阶段性回顾/治理启动时按此快速进入工作状态；具体规则判据仍以 [dev-rules.md](dev-rules.md) / [evidence-governance.md](evidence-governance.md) / 对应 `project-rules-<功能>.md` 为权威源。本文件与规则冲突时以规则为准。
>
> **来源**：从 2026-07-02 项目健康度审查四档执行闭环（7 commits / 56 项问题 / 5 根因）的实践提炼，作为审查报告 [project-health-audit-2026-07-02.md](project-health-audit-2026-07-02.md) 根因 4 暂缓项"阶段性收缩机制化"的参考载体（不作为规则强制）。
>
> **与规则的关系**：本文件不引入新规则，仅整合散落于各规则文件的阶段性动作，提供一份可线性阅读的操作流程。**引用维护机制**：本文件引用 dev-rules 等规则文件的具体小节号会随其重构而失效，§13 权威源索引为**单点维护点**——其他章节仅用语义描述（规则名称）引用，§号集中登记在 §13。若某处仍出现 §号硬引用，以 §13 为准。
>
> **参考 vs 规则的边界下沉声明**：下文出现的"N 步流程"（§5.2 / §6.2 / §7.3 / §8.2 等）均属**操作建议**，非强制规程；强制判据见对应引用的规则条款。读者不得将本文档的步骤序列作为"违反即触发 K 类问题"的依据——K 类问题针对的是规则硬约束违反，不是操作参考偏离。

| 是 | 不是 |

## 1. 阶段性治理的触发时机

阶段性治理**不是**定期例行公事，而是**特定触发条件命中**时启动的结构性收敛动作。

| 触发条件 | 来源 | 强度 |
|---------|------|------|
| 一个功能开发结束、开始下一个独立功能 | dev-rules 末尾"本文件的生命周期" | 强制（规则迁移点） |
| 定期文档健康度审查（默认月度，可降级） | dev-rules §6.5.1 | 强制 |
| 治理膨胀警报（单次审查 > 50 项高严重度） | dev-rules §6.5.2 | 强制 + 批量退役 |
| 不可抗力发生（外部环境非预期变化） | dev-rules §1.15 | 强制（阶段冻结） |
| ADR 状态大规模变更（≥3 份 active → superseded） | dev-rules §6.5.1 | 触发审查 |
| 用户手动要求 | dev-rules §6.5.1 | 强制 |

**禁止**：未命中任一触发条件时主动发起"治理整顿"——会陷入根因 1（用更多规则修没人执行的规则）陷阱。

## 2. 阶段定义

| 阶段 | 进入条件 | 退出条件 | 主要动作 |
|------|---------|---------|---------|
| **开发期** | §4 方向启动门控通过 + `project-rules-<功能>.md` 就位 | 功能冻结（核心 Verified + 文档闭环） | 源码开发 + 教训沉淀到项目级 + 顶层冻结 |
| **交界期** | 开发期结束 | 规则迁移完成 + handoff 重置 + 暂缓项归档 | 顶层解冻 + 规则提炼迁入 + 退役清理 |
| **审查期** | §6.5 触发条件命中 | 审查报告产出 + 改进项分档执行完 | 5 维度并行审查 + 四档优先级 + 严重度分级 |
| **阻塞期** | §1.15 不可抗力触发 | 7 步解除流程执行完 | 阶段冻结 + workaround + 监控恢复条件 |
| **暂缓期** | 改进项标 🕒 暂缓 | 触发评估条件命中（如下次审查） | 不主动推进，仅跟踪触发条件 |

**阶段可嵌套（通用规则）**：任意阶段遇不可抗力（dev-rules §1.15）均可被打断进入阻塞期，原阶段动作冻结，仅 workaround 推进；不可抗力解除后回到原阶段。典型示例：开发期中遇 codec bug → 阻塞期 → 解除后回开发期；审查期中遇审查工具失效 → 阻塞期 → 解除后回审查期。

## 3. 阶段启动门控

**权威源**：[dev-rules.md §4](dev-rules.md)——本节仅引用，不重复。

## 4. 阶段中的规则承载（K1-K3 教训内化）

> **来源**：审查报告 §11.5 K 类——分层治理硬约束违反。context-snapshot 开发期间 dev-rules 被修改 8 次（本应冻结），plugin/project-rules.md 治理类章节空缺。这不是"规则总量"问题，是"规则放错了层 + 硬约束未执行"。

### 4.1 顶层 vs 项目级的边界

| 层级 | 承载内容 | 修改约束 |
|------|---------|---------|
| **dev-rules.md**（顶层） | 跨功能通用规则 | 开发期**冻结**（§6 硬约束）；只在功能交界期迁入 |
| **evidence-governance.md**（顶层） | 跨功能证据规则 | 同上 |
| **reviewer-personas.md**（顶层） | 评审角色全集 + 触发条件 | 同上 |
| **`project-rules-<功能>.md`**（项目级） | 本功能开发期的所有新规则 | 开发期随发现随添加；功能结束后整体删除或提炼迁入 |

### 4.2 开发期规则承载纪律

1. **新规则先沉淀到项目级**：开发期发现的任何新规则，写入 `project-rules-<功能>.md §3` 治理类章节，**禁止**直接改 dev-rules
2. **顶层冻结的硬执行**：修改 dev-rules 前**必须**自检"本次修改是否在功能开发期" + "若在开发期，是否已走迁入流程"——违反即触发 K1 类问题
3. **项目级规则必备章节**（开发期补全，参考 [plugin/project-rules.md §3](plugin/project-rules.md)）：
   - P 级 handoff 触发器（项目专属 P 级定义）
   - 教训沉淀位置约定（哪些留项目级 / 哪些待迁顶层）
   - handoff schema 特化字段（如 blocker_ref / verified_evidence）
   - 项目级评审角色配置（从 reviewer-personas 全集选子集）
   - 跨会话续作读取门控（读哪些文件 + 门控判据）
   - 外部依赖版本变化重测流程（项目专属重测项）
   - 通用报错处理工作流（项目专属报错分类）

### 4.3 阶段中 handoff 规范

- handoff.md 末尾"新会话首动作"必须提示读 `project-rules-<功能>.md` §3（K3 教训）
- 未完成项三字段（id / confidence / depends_on）按 dev-rules §2.2 强制
- 项目特化字段（如 context-snapshot 的 blocker_ref / codec_status / verified_evidence）按 `project-rules-<功能>.md §3.3` 填写

## 5. 阶段结束收敛流程

开发期 → 交界期的过渡动作，按序执行：

### 5.1 收敛前置检查

- [ ] 核心 P0/P1 任务达到 Verified + 文档闭环
- [ ] 暂缓项已显式标注（不丢漏，不静默遗忘）
- [ ] 受阻塞项已登记 `depends_on` 指向阻塞源

### 5.2 收敛执行（7 步，操作建议）

> 边界声明：本节为操作建议，非强制规程；强制判据见 dev-rules §2（handoff 触发器）/ §6.4（规则退役）/ §6.5（审查退出）/ evidence-governance §19（Hypothesis 生命周期）。

| # | 步骤 | 完成检测方法 | 强制判据来源 |
|---|------|-------------|------------|
| 1 | **暂缓项归档**：将暂缓项汇总到 handoff.md 未完成项表，标 confidence = Hypothesis + 触发条件 | Grep `handoff.md` 未完成项表是否含所有 🕒 项 | dev-rules §2.2 三字段强制 |
| 2 | **Hypothesis 处置**：按 evidence-governance §19 检查超期 Hypothesis——超期降级为 Unknown，未超期登记补证计划 | Grep `Hypothesis` 在 docs/ 下，逐项核对 `声明日` + `必须补证日` | evidence-governance §19（强制） |
| 3 | **文档闭环**：核查 design.md / assessment.md 不可抗力声明块、review-closure-report.md 统计、ADR `evidence_as_of` / `expires_if_unchanged` | Read 上述文件对应字段，对比当前日期 | dev-rules §1.15 / evidence-governance §15（强制） |
| 4 | **机制候选清单核查**：mechanism-candidates.md 中已机制化 > 30 天的条目启动退休倒计时 | Read mechanism-candidates.md，对比 `已机制化日期` 字段 | dev-rules §6.4.4（强制） |
| 5 | **handoff 重置**：覆盖写 handoff.md，未完成项仅保留"需跨阶段跟踪"项；已完成项移到"本会话净变化"段 | Read handoff.md，确认未完成项表无已完成项 | dev-rules §2 / §2.2（强制） |
| 6 | **commit 序列化**：每档/每批独立 commit，便于回滚 | `git log --oneline -N` 确认 commit 数 = 档/批数 | dev-rules §2.1（强制 handoff commit） |
| 7 | **顶层解冻声明**：dev-rules.md 头部"🔴 冻结中"标注改为"🟢 解冻（功能交界期）" | Read dev-rules.md 头部，确认标注已更新 | dev-rules 末尾"本文件的生命周期"（强制） |

### 5.3 收敛退出判据

- handoff.md 未完成项表只含中/低优先级（无 P0/P1 遗留）
- 所有暂缓项有明确触发条件
- 无 Hypothesis 超期未处置
- 顶层规则文件状态标注正确（解冻中）

## 6. 规则迁移判据与流程

> **来源**：dev-rules 末尾"本文件的生命周期"3 段散文机制化。

### 6.1 迁移判据

一条项目级规则**满足下列任一条件**才迁入顶层：

| 判据 | 定义 | 示例 |
|------|------|------|
| **跨功能验证** | 在 ≥2 个独立功能开发期被触发且有效 | "契约优先（先读 .d.ts）"跨 plugin 通用 |
| **与具体功能无关** | 规则不含功能专属术语 / 路径 / API | "blocker_ref 字段"含 codec bug 专属 → 不迁 |
| **预期被下一功能复用** | 下一独立功能开发时大概率会用到 | "P 级 handoff 触发器"通用 → 迁（但 P 级定义留项目级） |

**禁止迁入**：
- 仅对当前功能成立的规则（如 context-snapshot 的 codec bug 处置流程）
- 含功能专属字段名的规则（如 `codec_status` / `verified_evidence`）
- 项目专属教训（如 L2 "死代码同模式复发"是 codec bug 特有）

### 6.2 迁移执行（5 步，操作建议）

> 边界声明：本节为操作建议，非强制规程；强制判据见 dev-rules 末尾"本文件的生命周期"（迁移时机）+ §6.4.2（退役引用清理清单）。

1. **筛选**：从 `project-rules-<功能>.md §3` 逐条评估是否满足 §6.1 判据
2. **改写**：去除功能专属术语，提炼为跨功能表述（如"context-snapshot 的 codec bug 阻塞"→"不可抗力阻塞"）
3. **写入顶层**：迁入 dev-rules.md 对应章节（不新建章节，融入既有结构）
4. **更新引用**：原 `project-rules-<功能>.md` 中该条改为指针引用顶层（"见 dev-rules §X.Y"）
5. **commit 标注**：commit message 含 `migrate: <规则名> project-rules → dev-rules`

### 6.3 不迁入的处置

未迁入的项目级规则随 `project-rules-<功能>.md` 整体删除（功能冻结后）。删除前**必须**：
- 确认无顶层规则引用该项目级规则（Grep 扫描）
- 确认无 ADR / handoff / investigation-note 引用
- 若有引用，先更新引用再删除

## 7. 阶段阻塞与恢复

**权威源**：[dev-rules.md §1.15](dev-rules.md)——本节仅引用流程框架。

### 7.1 阻塞触发

外部执行环境（VS Code 扩展 / CLI / SDK / 平台服务）可用性发生**非预期变化**时：
1. 在相关 design.md / assessment.md 插入不可抗力声明块
2. 依赖该环境的验证步骤标记"受限"或"推迟"
3. 登记待跟进项（GitHub issue / changelog 监控）+ 明确恢复条件

### 7.2 阻塞期动作

| 项分级 | 含义 | 处置 |
|--------|------|------|
| 🔴 搁置 | 必须该环境才能验证 | 暂停，等恢复 |
| 🟡 带观察推进 | 部分场景可避坑 | workaround + 标注"workaround ≠ 环境可用" |
| 🟢 可推进 | 不触发该环境缺陷 | 正常推进 |

### 7.3 恢复解除（7 步，操作建议）

> 边界声明：本节为 dev-rules §1.15 解除流程的摘要视图，**强制判据以 dev-rules §1.15 原文为准**。本节摘要若有偏差，以原文为权威。

按 dev-rules §1.15 解除流程执行：PR 合并 + release tag → 升级本地 CLI → 无 workaround 回归实测 → 移除声明 → 恢复搁置项 → 更新引用文档 → 归档 draft-issue。

**禁止**：
- 跳过回归实测直接移除声明
- 用 main 分支合并等同"已修复"（必须 release tag）
- 移除声明后不更新引用文档

## 8. 阶段性审查执行参考

> **来源**：本次 2026-07-02 审查实践提炼。下次审查可参考此流程快速启动。

### 8.1 审查范围

- `docs/` 全目录活跃文档（不含 archive/）
- 5 份规则文件覆盖率与重叠度
- `mechanism-candidates.md` 候选清单状态
- `handoff.md` 未完成项表推进情况

### 8.2 审查执行（5 维度并行子代理，操作建议）

> 边界声明：本节为审查执行的操作编排建议，非强制规程；强制约束见 verification-discipline skill（子代理纪律）+ dev-rules §6.5.1（审查范围）。

| 维度 | 职责 | 证据来源 | 完成检测方法 |
|------|------|---------|------------|
| 文档一致性 | 事实错误 / 内部矛盾 / 跨文档不一致 | 直接读文件 + 交叉对比 | 每条发现附文件路径 + 行号 + 矛盾对照 |
| 规则精简与边界 | 重复 / 冗余 / 层级错配 | 规则对比 | 每条发现附"规则 A §X 与规则 B §Y 重叠/冲突"对照表 |
| 研究方法论 | 模板执行 / 假设-验证闭环 | frontmatter 实读 | 每条发现附 frontmatter 字段缺失/过时清单 |
| 工作流与 SKILL | 触发-执行-退出全生命周期 | Grep 搜索调用记录 | 每条发现附 Grep query + 命中/零命中证据 |
| 知识沉淀 | 教训沉淀 / 文档时效 / 机制化治理 | 全库扫描 | 每条发现附文档清单 + 时效状态标注 |

**审查纪律**（[verification-discipline skill](../skills/verification-discipline/SKILL.md) 约束）：
- 子代理单源声明不升级为 Verified
- 停止条件：审查范围限定 docs/ 活跃文档，不扩到代码审查
- 目标相关：每条发现指向"下一步夯实基础"，不偏离到代码修复

### 8.3 严重度分级（下次审查生效）

| 级别 | 定义 | 处置 |
|------|------|------|
| **高-系统性** | 整类规则/机制失效 | 战略处置，需结构性改进 |
| **高-局部性** | 单处事实错误/矛盾 | 直接修正，1 次编辑 |
| **中** | 潜在风险或部分失效 | 排期修复 |
| **低** | 格式/链接/标注问题 | 批量处理 |

**禁止**：把 A1（笔误）和 C2（核心机制从未运行）都标"高"——压平严重度标尺会掩盖真正需要战略处置的少数项。

### 8.4 改进优先级四档（投入产出比排序）

> **级 vs 档的关系**：§8.3 的"严重度级"（高-系统性 / 高-局部性 / 中 / 低）决定**是否处置**——高级必处置，低级可批量或暂缓；§8.4 的"优先级档"（第一档 → 第四档）决定**处置顺序**——同级别内按投入产出比排序。两者正交：高级项可能落到任意档（如高-局部性的 A1 笔误落到第二档止血，高-系统性的 K1 项目级补全落到第一档减法）。

| 档 | 内容 | 原则 |
|----|------|------|
| **第一档：减法 + 结构性补全** | 删除冗余规则 + 降级零执行角色 + 补项目级空缺 | 缩小系统面积优先于加新规则 |
| **第二档：止血** | 事实修正 + ADR 时效 + Hypothesis 复查机制 | 事实过时拖越久失真越严重 |
| **第三档：机制缺口** | 工作流退出流程 + 子代理证据规则 + 评审闭环 | 系统瘦身后再做流程设计 |
| **第四档：元规则** | 退休机制 + 审查退出条件 + 时效模型 | 验证精简体系能运转后再叠加 |

**分批 commit**：每档独立 commit，每档内部可分批（本次第四档分 3 批）。commit message 格式：`docs: 第N档<摘要> - <内容>`。

### 8.5 根因分析

每条发现归到方法论根因（≥1 条）。本次 5 根因：

| 根因 | 表现 | 改进原则 |
|------|------|---------|
| 1. 规则制定无"可执行性评估" + 层级错配 | frontmatter 零执行 / 角色零调用 / 硬引用将失效 | 每条规则含"执行证据"+"检测方法" + 评估层级归属 |
| 2. 知识沉淀无"生命周期管理" | 模板执行不一致 / Hypothesis 无跟踪 / 机制无退休 | 每个产物定义"声明日+复查日+退役条件" |
| 3. 工作流设计"重触发轻退出" | handoff 读无门控 / §1.15 解除流程缺失 / 70% 不可操作 | 每个工作流定义"触发→执行→退出"全生命周期 |
| 4. 治理预算无约束（层级错配症状） | 顶层该冻结没冻结 + 项目级该填充没填充 | 先执行已有硬约束，再评估 one-in-one-out |
| 5. 规则演进"加法 only" | ADR 只增不减 / 规则只加不改 / 角色只立不废 | 补规则退役流程 + 引用清理清单 |

### 8.6 暂缓项管理

暂缓项**必须**显式标注，禁止静默遗忘：

| 字段 | 要求 |
|------|------|
| 触发条件 | 何时重新评估（如"下一会话"/"codec bug 修复后"/"下次审查"） |
| confidence | 通常 Hypothesis（暂缓意味着未验证） |
| depends_on | 若受阻塞，指向阻塞源 ID |
| 归档位置 | handoff.md 未完成项表 + 审查报告暂缓清单 |

### 8.7 审查退出条件

按 [dev-rules §6.5.2](dev-rules.md) 执行——审查机制自身不假定永久运行：

| 条件 | 动作 |
|------|------|
| 连续两次高严重度 < 10 | 月度降季度 |
| 连续两次高严重度 < 3 | 季度降半年度 |
| 单次 > 50 | 治理膨胀警报（强制批量退役） |
| 连续 6 月未执行 | 审查机制退役，转按需触发 |

## 9. 阶段切换的 handoff 规范

### 9.1 何时写 handoff

**权威源**：[dev-rules §2](dev-rules.md)——本节仅引用。

- 触发器 a：用户口头要求（无条件）
- 触发器 c：对话过长 + 话题跳转 + 上下文吃紧（三条 AND，仅提议不自动）

**禁止**：未命中触发器主动写 handoff（如刚完成上一份 handoff 的"下次具体动作"第一项 ≠ 触发器）。

### 9.2 handoff 三字段强制

未完成项表**必须**含三字段（[dev-rules §2.2](dev-rules.md)）：

| 字段 | 规则 |
|------|------|
| `id` | 稳定短 ID，小写+数字+连字符，≤20 字符 |
| `confidence` | Verified / Likely / Hypothesis（与 evidence-governance §4 一致） |
| `depends_on` | 指向其他稳定 ID，**禁止**自然语言描述 |

### 9.3 阶段切换 handoff 模板

```markdown
# Handoff — <阶段名>

> **生命周期**：每次覆写，无归档——会话交接文件。

## 本会话决策
| id | 决策 | confidence | depends_on |

## 本会话净变化
### 1. <commit 摘要>（commit `<hash>`，N files）
- ...

## Commits
| Hash | Repo | Message |

## 未完成项
| id | 方向 | 说明 | 优先级 | confidence | depends_on |

## 权威源
[规则文件链接列表]

> **新会话首动作**：除读 dev-rules.md + handoff.md 外，必须读
> [project-rules-<功能>.md](project-rules-<功能>.md) §3 治理类规则。

---
## Handoff
<一句话摘要，供用户在新会话开头粘贴>

**后续动作**（按优先级）：
1. 🟡 中：...
2. 🟢 低：...
```

### 9.4 handoff 进入 git

每次覆盖写 handoff.md 后**立即** git commit（[dev-rules §2.1](dev-rules.md)）。commit message：`handoff: <一句话摘要>`。

## 11. 成熟实践映射

| 本文件机制 | 映射实践 | 说明 |
|-----------|---------|------|
| 阶段定义（开发期/交界期/审查期/阻塞期/暂缓期） | **PDCA 循环** + **SRE SLO 错误预算**（见下行"审查退出条件"复用 SRE 错误预算） | Plan/Do/Check/Act 对应开发/收敛/审查/改进；阻塞期对应错误预算耗尽冻结变更 |
| 四档优先级（减法→止血→机制→元规则） | **技术债偿还优先级**（SonarQube） | 高息债先还（结构性缺口）→ 逾期账单（事实过时）→ 长期投资（机制设计）→ 治理改革（元规则） |
| 顶层 vs 项目级分层 | **联邦制治理**（Federal Governance） | 中央（顶层）只管跨州通用；州（项目级）管本地专属；硬约束双向（中央不得越权 + 州不得空缺） |
| 暂缓项显式标注 | **Tech Debt Backlog** | 禁止静默遗忘，必须显式登记 + 触发条件 + 依赖关系 |
| 审查退出条件 | **SRE SLO 错误预算**（与阶段定义行复用）+ **PDCA Check 退出判据** | 预算充裕放宽节奏，预算耗尽冻结；Check 不必无限循环，稳定后转 Act 维护 |
| 规则迁移判据 | **代码提取公用函数的判据**（Rule of Three） | 跨 ≥2 个功能验证才迁入"公共库"（顶层） |
| commit 序列化（每档独立） | **Atomic Commit** + **Git 回滚单元** | 每档独立 commit 便于异常时回滚到任一档快照 |
